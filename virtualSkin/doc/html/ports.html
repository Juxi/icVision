YARP Ports of the Reflex Demo

Remote Control Board

The Remote Control Board is YARP's interface to a motor, and to each instance belongs three YARP ports:

  • /NameOfYourRemoteControlBoard/command:i
  • /NameOfYourRemoteControlBoard/state:o
  • /NameOfYourRemoteControlBoard/rpc:i

For each control board interface, Virtual Skin creates a trio of ports and connects them as shown:

The iCub robot has 6 such control boards that correspond to its body parts: torso, head, right_arm, left_arm, right_leg and left_leg.

These create the following motor control ports:

  • /icub/torso/command:i
  • /icub/torso/state:i
  • /icub/torso/rpc:i
  • /icub/head/command:i
  • /icub/head/state:i
  • /icub/head/rpc:i
  • /icub/right_arm/command:i
  • /icub/right_arm/state:i
  • /icub/right_arm/rpc:i
  • /icub/left_arm/command:i
  • /icub/left_arm/state:i
  • /icub/left_arm/rpc:i
  • /icub/right_leg/command:i
  • /icub/right_leg/state:i
  • /icub/right_leg/rpc:i
  • /icub/left_leg/command:i
  • /icub/left_leg/state:i
  • /icub/left_leg/rpc:i

The robot models provided with this distribution of Virtual Skin include only the iCub's upper body. Therefore, filters are only created for the control boards: torso, head, left_arm and .

This results in the following filtered ports:

  • /icubF/torso/command:i
  • /icubF/torso/state:i
  • /icubF/torso/rpc:i
  • /icubF/head/command:i
  • /icubF/head/state:i
  • /icubF/head/rpc:i
  • /icubF/right_arm/command:i
  • /icubF/right_arm/state:i
  • /icubF/right_arm/rpc:i
  • /icubF/left_arm/command:i
  • /icubF/left_arm/state:i
  • /icubF/left_arm/rpc:i

As you can see, the names of the filtered ports reflect the names of the unfiltered iCub motor control ports, however an "F" has been appended to the robot name to signify that the port is filtered. The filtered ports allow transparent access to the underlying iCub motor control ports. Therefore existing tools in the iCub repository, such as robotMotorGui, which connect to a set of ports based on a robot name, can be connected to the filtered ports by simply requesting the robot "iCubF". Such tools will then function as usual, however their access to the robot (or simulator) will be controlled by the filter, which may intervene in either of these two ways:

  1. The filter can cut the connection
  2. The filter may inject
